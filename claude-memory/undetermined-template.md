# 概述

本文档用于描述待定内容的规范模板

# 格式要求

## 基本规范
1. **评分制**：使用-100到100的分数评分，而不是百分比
2. **独立建议**：每个待定项必须是独立的"建议采纳"或"建议去除"
3. **多方案处理**：同一问题的不同解决方案要分别创建不同编号的待定项
4. **排序规则**：默认把打分最高的可能目的排在最后
5. **直接描述**：优化后的内容中直接描述具体方案，不要列举"可能的实现方案"

## 编号规范
- 格式：`<待定项编号>.<方案编号>`
- 示例：对于路径接收方式问题
  - `002.001`：仅命令行参数方案
  - `002.002`：命令行+拖拽方案  
  - `002.003`：多种方式方案
  - 按评分从低到高排序，最高分的放在最后

# 模板

```
# ✅ <待定项编号>.<方案编号> 建议采纳 - <分数> ：

- 原文：（<原文件相对于项目根目录的路径>， <原文所在行数>）

> 原文内容

- 优化后的内容：

> 具体的优化方案描述

- 权衡点：
  - 加分项（建议采纳 + <分数>）：……
  - 加分项（建议采纳 + <分数>）：……
  - 减分项（建议去除 - <分数>）：……
  - ……
```

或

```
# ❌ <待定项编号>.<方案编号> 建议去除 - <负分数> ：

- 原文：（<原文件相对于项目根目录的路径>， <原文所在行数>）

> 原文内容

- 优化后的内容：

> 具体的优化方案描述（如删除该内容或替换为其他内容）

- 权衯点：
  - 减分项（建议去除 - <分数>）：……
  - 加分项（建议采纳 + <分数>）：……
```

# 示例

> 假设当前工作目录在/path/to/workspace，有一个用于描述任务的文件是/path/to/workspace/tasks/task-1.md。
>
> 然后对其进行分析和优化，然后将优化后需要询问用户的待定项写入到/path/to/workspace/claude-memory/undetermined.md。

`/path/to/workspace/claude-memory/undetermined.md`的内容应该类似下面所示：

```markdown
# ✅ 001.001 建议采纳 - 85 ：

- 原文：（tasks/task-1.md， 15-17）

> （文内容第15行，含有目标原文的部分）
> （文内容第16行）
> （文内容第17行，含有目标原文的部分）

- 优化后的内容：

> 消除冗杂表述的具体优化方案

- 权衡点：
  - 加分项（建议采纳 + 60）：消除冗杂，提高可读性
  - 加分项（建议采纳 + 30）：表述更加简洁清晰
  - 减分项（建议去除 - 5）：可能丢失部分细节
  
# ✅ 002.001 建议采纳 - 60 ：

- 原文：（tasks/task-1.md， 20）

> 接收路径参数

- 优化后的内容：

> 通过命令行参数接收路径

- 权衡点：
  - 加分项（建议采纳 + 70）：实现简单，符合标准CLI工具惯例
  - 减分项（建议去除 - 10）：功能相对单一

# ✅ 002.002 建议采纳 - 80 ：

- 原文：（tasks/task-1.md， 20）

> 接收路径参数

- 优化后的内容：

> 支持通过命令行参数和Windows拖拽两种方式接收路径

- 权衡点：
  - 加分项（建议采纳 + 50）：兼顾命令行和图形界面用户习惯
  - 加分项（建议采纳 + 40）：与拖拽要求呼应
  - 减分项（建议去除 - 10）：实现复杂度适中

# ✅ 002.003 建议采纳 - 90 ：

- 原文：（tasks/task-1.md， 20）

> 接收路径参数

- 优化后的内容：

> 支持多种路径输入方式：命令行参数、Windows拖拽、交互式输入

- 权衡点：
  - 加分项（建议采纳 + 70）：用户体验最佳，适应各种使用场景
  - 加分项（建议采纳 + 30）：功能完整全面
  - 减分项（建议去除 - 10）：实现复杂度较高

# ❌ 003.001 建议去除 - -30 ：

- 原文：（tasks/task-1.md， 25）

> 某个冗余的功能描述

- 优化后的内容：

> 删除该冗余描述

- 权衡点：
  - 减分项（建议去除 - 40）：功能冗余，增加复杂度
  - 加分项（建议采纳 + 10）：可能在某些场景下有用

(其他略……)
```
